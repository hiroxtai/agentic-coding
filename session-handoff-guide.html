01

なぜ引き継ぎが必要か

エージェントの記憶はセッションごとに消える。引き継ぎとは「窓の外に記憶を残し、次が読み戻す」設計のこと。

長時間・複数日に渡る開発で最大の壁は、各セッションが前回の記憶ゼロで始まることです。Anthropic はこれを「シフト制で働くエンジニアが、毎回前のシフトの記憶なしで現れる」と喩えました。Cline の Memory Bank も「私の記憶はセッション間で完全にリセットされる。だからこそ完璧な文書化に頼る」という同じ前提から出発します。

つまり「セッション間引き継ぎ」とは、コンテキスト窓という揮発メモリの外に、永続的な記憶を構造化して書き残し、次のセッションがそれを読んで現在地を復元する仕組みづくりにほかなりません。2026年には、これが「個別の工夫」から独立した設計領域へと成熟し、専用ツールキット・MCP基盤・ベンチマークまで整いました。

◆ 前回ガイドとの接続
本資料は「コーディングエージェント開発手法ガイド」07章（ADR/メモリ駆動の引き継ぎ）を、独立した1テーマとして深掘りするものです。05章で扱った Spec Kit が「仕様」を中心に置くのと対をなし、ここでは「記憶」を中心に置く仕組みを扱います。
02

全体像：3層モデル

引き継ぎの仕組みは独立した3つの層の組み合わせ。混同しがちだが、役割は直交している。

世の中のツールや手法は一見ばらばらですが、「何を書くか（規約）」「どう保存・検索するか（基盤）」「エージェントがどう使うか（組込み）」の3層に整理すると見通せます。それぞれ別々に選べるので、要件に応じて足し引きできます。

✔ 汎用化のカギ
「様々なプロジェクトで使える」ことを最優先するなら、まず規約層（テキストファイルの約束事）を固めるのが正解です。これはツール非依存で git に乗り、人もレビューできます。基盤層（MCP）は後から重ねられるので、最初から作り込まないこと。
03

Memory Bank（最有力の対応物）

Spec Kit が「仕様」を中心に置くように、Memory Bank は「プロジェクトの記憶」を構造化Markdownで外部化する方法論。

Memory Bank（Cline 発祥、Roo Code 版もあり）は、ステートレスなエージェントを「永続的な開発パートナー」に変えるドキュメント方法論です。memory-bank/ フォルダに、互いに積み上がる階層でMarkdownを配置します。下の図がその依存関係です。

仕組みと運用

運用ルールは .clinerules/memory-bank.md（カスタム指示）に書きます。中心となる鉄則は「毎タスク開始時に全ファイルを必ず読む（任意ではない）」。主なコマンドは次の3つです。

  • initialize memory bank ― 新規プロジェクトで初期構造を作る。
  • update memory bank ― 全ファイルを見直して更新（特に activeContext と progress）。
  • follow your custom instructions ― Memory Bank を読み、中断地点から再開する。

窓が埋まってきたら /newtask や /smol で会話を「plan / 済んだ作業 / 関連ファイル / 次の手」に蒸留し、クリーンな窓で続行します。

◆ あなたの目的に直結する最大の利点：ツール非依存
Cline 公式が明言する通り、Memory Bank は「ドキュメントを読めるAIなら何でも動く」方法論です。コマンド名こそ違えど、Claude Code・Cursor・Roo Code などにそのまま流用できます。README との違いは、頻繁に変わる「現在の文脈」と「進捗」を専用ファイルで追跡する点にあります。

05

設計の核：メモリの3種類

「何を引き継ぐか」は3種類に分かれる。コーディングでは特に procedural（手続き的記憶）が効く。

優れた引き継ぎ基盤を設計するには、記憶を3種類に分けて考えるのが定石です。多くのシステムは episodic と semantic の2つに注力しがちですが、本番のコーディングエージェントには第3の procedural（手続き的記憶）が不可欠だと指摘されています。

06

マルチスコープと多信号検索

記憶を「誰の・どの範囲か」で仕分け、検索は複数の信号を融合する ― これが本番品質の2大設計。

記憶基盤を自作・選定するなら、スコープと検索の設計が肝心です。Mem0 が確立し定着した4スコープモデル（user / agent / session / org）は、横断利用と混線防止を両立させます。検索側は単一のベクトル類似だけに頼らず、意味・キーワード・エンティティを並列に評価して融合するのが2026年の標準です。

▶ 本番運用で効く小技
記憶の書き込みは非同期（応答を遅らせない）、検索後にリランク（順序を整える）、メタデータでフィルタ（プロジェクトや期間で絞る）。これらが「実験」と「本番」を分ける実務的な差です。
07

推奨アーキテクチャ：二段構え

汎用・横断で自作するなら、「ファイル規約を土台に、必要に応じてMCPを重ねる」二段構えが現実解。

「様々なプロジェクトで使える」を最優先するなら、可搬で透明なファイル規約を土台に置き、検索・自動化が要る段階でMCPメモリ層を重ねるのが2026年6月時点の最適解です。役割が直交するので、小さく始めて後から足せるのが利点です。

✔ なぜこの二段構えなのか
ファイル層はツールを変えても生き残り、人もエージェントも読め、git 履歴で監査できます。MCP層は「複数ツールで同じ記憶を共有したい」「自動抽出・検索が欲しい」段階で初めて効きます。最初からMCPを作り込まないのが、汎用性とコストの両立の鍵です。
08

セッションのライフサイクル

仕組みを「運用ルール」に落とすと、毎セッションがこの循環になる。AGENTS.md とフックで自動化する。

⚠ 自動化のポイント
人間が毎回手で指示すると形骸化します。Claude Code ならhook（決定的に必ず実行）やskill（/handoff のように呼ぶ）、Cline なら .clinerules に固定するのが堅実。「より良いセッションを、より良い“仕組み”に変える」のが要点です。
09

未解決問題と注意点

基盤を自作するなら、ここを運用ルールで補う必要がある。2026年でもまだ“開いた”問題。

メモリ分野は成熟しましたが、残る難問は「具体的で境界は明確だが、根本解決には至っていない」ものばかりです。汎用基盤を作る側は、これらを認識して運用で補う必要があります。

問題何が起きるか運用での補い方
陳腐化
(staleness)		よく参照される記憶が、状況変化後に「自信満々に間違い」になる（例：転職後も古い所属を返す）。タイムスタンプ・出典を必ず残し、定期的に update。事実には鮮度を併記。
進化 vs 上書き多くのシステムは変化を「置換」として扱うが、本来は「変遷」として理解すべき。ADR を追記式にし、過去の決定を消さず「superseded」で残す。
時間的抽象化文脈が大規模化するほど時間推論が崩れる（ベンチでも最難関カテゴリ）。進捗は時系列で、節目に要約。古い詳細は畳んで概要だけ残す。
同一性の解決user_id 前提が崩れる（匿名・複数端末・混在認証）。スコープIDをアプリ側の認証に紐づけ、メモリ層で生成しない。
プライバシー/同意誰が記憶を閲覧・保持・削除できるかはアプリ層の設計事項。保持期間・削除手段・閲覧範囲を最初から決め、ローカル保存も選択肢に。
⚠ ベンチマーク値の扱い
各ツールの性能（LoCoMo / LongMemEval / BEAM などのスコア）や順位には自己申告を含み、用いるベンチも異なります。Mem0 自身も推奨する通り、基盤を確定する前に自分のワークロードで評価フレームを回すのが安全です。トークン効率（1クエリあたりの消費）も本番のコストに直結するため要確認です。

使い分け・出発点

「小さく始め、必要になったら足す」。状況別の最初の一歩。

あなたの状況最初の一歩理由
まず1プロジェクトで手堅くMemory Bank をコピー最速・無料・ツール非依存。git に乗り、すぐ使える。
複数ツール横断で記憶共有OpenMemory MCP（ローカル）Cursor↔Claude↔Windsurf を跨いで同一記憶。約5分で導入。
Claude Code 中心CLAUDE.md / AGENTS.md ＋ progress ＋ skill/hook公式路線（構造化ノート・長時間ハーネス）と整合し自動化しやすい。
引き継ぎをスキルとして自動化agentmemory の handoffCline / Claude Code に自動組込でき、呼ぶだけで蒸留・引き継ぎ。
◆ 通底する4原則
①記憶は窓の外（ファイル）に置く ― ツールを変えても生き残る。②開始時に必ず読む ― 任意にしない。③区切りで蒸留して書き戻す ―「plan/done/files/next」。④決定は“なぜ”ごと残す ― ADRで蒸し返しを防ぐ。ツールはこの4つの自動化手段に過ぎません。
