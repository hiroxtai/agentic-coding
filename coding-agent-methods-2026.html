01

全体像 ― なぜ「手法」が必要なのか

手法の違いは「好み」ではなく、エージェントの制約への対処法の違いである。

2026年6月時点で、Claude Code をはじめとするコーディングエージェントは「質問に答えて待つチャットボット」から「ファイルを読み、コマンドを実行し、自律的に問題を解決する実行環境」へと位置づけが変わりました。あなたは自分でコードを書く代わりに「何が欲しいか」を記述し、エージェントが探索・計画・実装を行う──この前提の転換が、新しい開発手法を次々に生んでいます。

一見ばらばらに見える各手法（プランモード、仕様駆動、ADR駆動、最小コンテキスト、長時間タスク…）は、実はたった一つの制約への異なる対処として理解できます。それが「コンテキストウィンドウは有限であり、埋まるほど性能が劣化する」という事実です。Anthropic の公式ベストプラクティスも、ほとんどの推奨事項はこの一点に基づくと明言しています。

◆ この資料の読み方
まず 「土台（02章）」 でエージェント共通の制約を押さえ、次に 6つの中核手法（04〜09章） を「単一セッション系 → 引き継ぎ系 → スケール系」の順に見ていきます。最後に 横断的なベストプラクティス（10章） と 使い分けの指針（11章） でまとめます。
02

土台：コンテキストエンジニアリング

すべての手法が立脚する共通原理。プロンプトエンジニアリングの次に来た考え方。

Anthropic は2025年9月の記事で、コンテキストエンジニアリング（context engineering）を「プロンプトエンジニアリングの自然な進化形」と定義しました。プロンプト（指示文）だけでなく、システムプロンプト・ツール定義・MCP・外部データ・会話履歴を含むコンテキスト全体を、推論のたびに最適にキュレーションする技術です。

なぜ「少ないほど良い」のか ― 注意の予算とコンテキストの腐敗

LLM は人間と同じく、扱う情報が増えると焦点を失います。トークン数が増えるほど情報の正確な想起能力が落ちる現象は 「コンテキストの腐敗（context rot）」 と呼ばれます。これは Transformer の構造（n個のトークンで n² の関係を扱う）に由来する、すべてのモデルに共通する性質です。つまりコンテキストは「使えば減る有限の注意の予算（attention budget）」として扱うべきものです。

したがって、良いコンテキスト設計とは「望む結果を得る確率を最大化する、最小の高シグナルなトークン集合を見つけること」です。ただし「最小＝短い」ではありません。必要な情報は十分に与えたうえで、ノイズを削るという意味です。この「ちょうど良い高度（right altitude）」の感覚が、後述するすべての手法の判断軸になります。

長時間タスクを支える3つの基本テクニック

1回のコンテキスト窓を超える長い作業（大規模移行、包括的な調査など）のために、Anthropic は3つの技術を挙げています。これらは04章以降の具体的手法の「部品」になります。

テクニック内容得意な場面
コンパクション
Compaction		窓が一杯に近づいたら会話を要約し、新しい窓を要約から再開。Claude Code は重要な設計判断・未解決バグ・実装詳細を残し、冗長なツール出力を捨てる。往復の多い対話的タスク

03

手法マップ（俯瞰図）

6つの中核手法は「土台の上に積み上がり、横断的な実践に支えられる」という構造で理解できる。

✔ 重要な前提
これらは「どれか1つを選ぶ」ものではありません。たとえば「プランモードで仕様を固め（04）、SPEC.mdに書き出し（05）、新セッションで最小コンテキストで実装し（06）、進捗ファイルで引き継ぐ（07）」のように連結するのが典型的な使い方です。
04

探索→計画→実装→コミット（プランモード）

Anthropic公式が推奨する最も基本的な4フェーズ・ワークフロー。「間違った問題を解く」事故を防ぐ。

エージェントにいきなりコードを書かせると、「間違った問題を解いたコード」が生まれがちです。そこで Claude Code のプランモード（読み取り専用）を使い、探索と実行を分離します。公式が勧める流れは次の4段階です。

仕様を「インタビュー」で詰める応用

大きめの機能では、最小限のプロンプトから始めてエージェントに逆質問させるのが有効です。公式は次のようなプロンプトを推奨しています。

「[簡単な説明] を作りたい。AskUserQuestion ツールで私に詳しくインタビューして。技術実装・UI/UX・エッジケース・トレードオフについて、私が見落としている難所を掘り下げて。すべて網羅したら SPEC.md に完全な仕様を書き出して」

仕様が完成したら新しいセッションを開始して実装に移ります。新セッションは実装だけにクリーンなコンテキストを使え、参照すべき仕様書も手元にある──これが05章の仕様駆動開発への橋渡しになります。

⚠ プランモードを使わない方が良い場面
計画にはオーバーヘッドがあります。スコープが明確で修正が小さいとき（タイポ修正、ログ追加、変数リネーム）は直接やらせるべきです。公式の目安は「差分を1文で説明できるなら計画は省略する」。計画が活きるのは、アプローチが不確実・複数ファイルに及ぶ・対象コードに不慣れな場合です。
05

仕様駆動開発（Spec-Driven Development）

コードではなく「実行可能な仕様」を真実の源にする。2026年に主要ツールが軒並み採用した方法論。

仕様駆動開発（SDD）は、コードではなく詳細な仕様こそが真実の源（source of truth）とする方法論です。まず「システムが何をすべきか」を記述した仕様を書き、そこから実装計画を導き、原子的なタスクに分解し、最後にコードを生成します。2025年に「vibe coding」（意図から逸れた・APIを幻覚する・規模とともに腐るコードを量産する失敗モード）への対抗として登場しました。

3つの実装レベル

レベル仕様とコードの関係位置づけ
spec-first仕様が生成を駆動し、コードは保守対象として残る導入しやすい標準形
spec-anchored仕様とコードが共存し、相互に整合を取る中間形
spec-as-source仕様が唯一の成果物、コードは完全に生成される最も先鋭的

2026年のツール状況

2026年までに主要なAIコーディングツールがそれぞれの流儀でSDDを実装しました：GitHub Spec Kit（オープンソースのツールキット）、AWS Kiro、Claude Code、Cursor、OpenSpec、BMAD、Tessl、Google Antigravity など。Red Hat や Microsoft が企業向けSDDガイドを公開する段階に入っています。AWS Kiro は「仕様を先に書いたことで、40時間規模の機能を人間8時間未満で出荷」した顧客事例を文書化しています。

▶ Claude Code での実践
Claude Code には専用の「SDDモード」コマンドはありませんが、04章の「インタビュー → SPEC.md」がそのまま軽量なSDDになります。公式は「最も有用な仕様は自己完結的で、関係するファイルとインターフェースを名指しし、スコープ外を明記し、機能が動くことを証明するE2E検証ステップで締める」と述べています。
06

最小コンテキスト指示（余計な情報を伝えない）

「賢いモデルには、指示しすぎない」。能力が上がるほど、過剰な指示はむしろ害になる。

「余計な情報は一切伝えずに開発を指示する」という手法は、02章のコンテキストエンジニアリングの直接の帰結です。Anthropic は「モデルが賢くなるほど、規定的なエンジニアリングは少なくて済み、エージェントはより自律的に動ける」と述べ、まず最小プロンプトを最良モデルで試し、失敗モードが見つかったときだけ指示や例を足していく進め方を勧めています。

「ちょうど良い高度」を狙う

システムプロンプトには失敗の両極があります。片方は複雑で脆いロジックをハードコードしすぎる極、もう片方は曖昧すぎて具体的な信号を欠く極。その中間が「ちょうど良い高度」です。

なぜ効くのか ― 研究も裏づける

  • クロス汚染を防ぐ：各ワーカーは「自分の仕事に必要なことだけ」を知る。ユーザー履歴や他エージェントの状況を渡さないことで、コンテキストが軽く保たれ、デバッグも容易になる（最小共有コンテキストの原則）。
  • 重要な制約が埋もれない：コンテキストは「テキスト量」ではない。有用なのは「良い答えの形を変える小さな詳細集合」。情報過多はそれを希釈し、モデルが鍵となる制約を無視する確率を上げる。
  • TDDでの逆説的な事例：あるSKILL.mdを「107行の詳細な9フェーズTDD指示」から「20行の簡潔な指針（修正・grep・検証）」へ削ったところ、解決率が12%→50%へ約4倍に向上したという報告もある。指示しすぎは野心的すぎる修正と巻き込み事故を招いた。
⚠ 「伝えない」の正しい意味
これは「丸投げ」ではありません。必要な制約・参照先・検証基準は明確に与えたうえで、モデルが自力で導ける情報（コードを読めば分かること等）を渡さないという意味です。CLAUDE.md に「コードを読めば分かること」を書くと、かえって本当に必要な指示が無視される、と公式も警告しています。
07

セッション間引き継ぎ（ADR / メモリ駆動）

「毎シフト、記憶のない新人が来る」問題への対処。決定と進捗を外部に書き残し、窓を越えて継承する。

長時間エージェントの核心的な課題は、各セッションが前回の記憶ゼロで始まることです。Anthropic はこれを「シフト制で働くエンジニアが、毎回前のシフトの記憶なしで現れる」と喩えています。対処は一貫して「コンテキスト窓の外に成果物を書き残し、次のセッションが読み戻す」こと。その成果物には複数の種類があります。

4種類の引き継ぎ成果物

成果物役割2026年の位置づけ
CLAUDE.md /
AGENTS.md		毎セッション冒頭で読まれる永続コンテキスト。ビルド/テストコマンド、コード規約、リポジトリ作法など「コードから推論できないこと」を記す。AGENTS.md は6万以上のリポジトリが使うツール横断の標準に。/initで雛形生成。
ADR /
AgDR		アーキテクチャ上の決定と「なぜ」を記録。AIが過去の決定を再蒸し返すのを防ぐ。AgDR（Agent Decision Records）はAIの選択にも同じ厳密さを与える拡張。「AIが決めたこと」も人間と同じ追跡可能性で残す動きが拡大。
進捗ファイルclaude-progress.txt / NOTES.md に「何をやったか」を逐次記録。02章の構造化ノートの具体形。長時間ハーネス（08章）の必須部品。
git 履歴説明的なコミットメッセージ＝復帰可能な状態。悪い変更を revert でき、次のagentが履歴で経緯を追える。「動く状態」を保証する命綱。
⚠ CLAUDE.md は「短く」保つ
公式の鉄則：各行について「これを消したらClaudeが間違えるか？」と問い、Noなら消す。肥大化したCLAUDE.mdは、本当に重要なルールがノイズに埋もれて無視される原因になります（06章と同じ原理）。頻繁に変わる情報・APIの詳細仕様は書かず、リンクで参照する。
08

長時間タスク（Long-Running Agents）

数時間〜数日のタスクを一気に走らせる。鍵は「初期化役」と「実装役」への役割分離と、1機能ずつの漸進。

Anthropic は2025年11月、Claude Agent SDK で多数のコンテキスト窓をまたいで一貫して進捗するためのハーネス（足場）を公開しました。コンパクションだけでは不十分で、高レベルなプロンプト1つでは最前線モデルでも本番品質のWebアプリを作り切れない、という観察が出発点です。失敗は2パターン現れました。

  • 一度に全部やろうとする（ワンショット狙い）→ 実装の途中でコンテキストが尽き、次のセッションが中途半端な状態を引き継ぐ。
  • 早すぎる「完成宣言」 → 後続のagentが「進捗あり」を見て、まだ途中なのに完了とみなす。

解決策は2つの役割への分離でした（※システムプロンプトやツールは同一で、最初のユーザープロンプトだけが異なる）。

失敗モードと対策（公式まとめ）

問題初期化エージェントの対策実装エージェントの対策
プロジェクト全体を早く完成宣言してしまう仕様からE2E機能リスト(JSON)を作る冒頭でリストを読み、着手する1機能を選ぶ
バグや未文書の状態で環境を放置する初期gitリポジトリと進捗ノートを用意冒頭で進捗とgit logを読み、サーバで基本テスト。終了時にcommitと進捗更新
機能を早まって「完了」にする機能リストを用意全機能を自己検証。慎重なテスト後のみ「passing」に
アプリの起動方法を毎回探るinit.sh を書く冒頭で init.sh を読む
▶ 実装の工夫メモ
機能リストはMarkdownより JSON が良いとされます（モデルがJSONを勝手に書き換えにくいため）。テストは curl やユニットテストだけでなく、ブラウザ自動化で人間と同じE2E検証をさせると劇的に精度が上がる、と報告されています。
09

マルチエージェント・動的ワークフロー

1つの問題を多数のサブエージェントに分割し並列処理。2026年5月の「動的ワークフロー」で実用段階に。

02章のサブエージェントを推し進めると、1つのリードエージェントが高レベルな計画を持ち、複数のサブエージェントが並列に深掘りして要約を返すマルチエージェント構成になります。さらに2026年5月、Claude Code は動的ワークフロー（Dynamic Workflows）を発表しました。Claude がオーケストレーションのスクリプトを自分で書き、数十〜数百の並列サブエージェントを1セッション内で走らせ、結果を統合前に検証します。

動的ワークフローの要点（2026年5月時点）

  • 規模：1回の実行で同時最大16・累計最大1,000のサブエージェント。数時間〜数日に及ぶ長時間・並列作業向け。
  • 中断耐性：進捗が逐次保存され、中断しても最初からではなく途中から再開する。
  • 起動方法：「ワークフローを作って」と頼むか、新設定 ultracode（努力度を最高に設定し、Claudeが自動でワークフロー化を判断）をオンにする。auto モード推奨。
  • 用途：コードベース全体のバグ狩り・最適化監査・セキュリティ監査、数千ファイルに及ぶ大規模移行、二重チェックが必要な重要作業。
  • コスト注意：通常セッションより大幅にトークンを消費。まずは小さくスコープしたタスクで感触を掴むのが推奨。Max / Team / Enterprise（管理者有効化）のリサーチプレビュー。
◆ 実例：Bun の Zig → Rust 移植
動的ワークフローで、Bun を Zig から Rust へ移植。既存テストの99.8%が通過、約75万行のRust、初コミットからマージまで11日。あるワークフローがZig側の全構造体フィールドに正しいRustのライフタイムを対応づけ、次のワークフローが各 .zig を挙動同一の .rs へ移植（各ファイルに2名のレビュー役つきで数百エージェントが並列）。「四半期で計画する規模の作業が数日で終わる」段階に入っています。

基礎となる5つのワークフローパターン（Building Effective Agents）

これらの上位構成は、Anthropic が2024年末に整理した5つの基本パターンの組み合わせとして理解できます。設計の3原則は「単純さを保つ・計画ステップを明示して透明性を高める・ツール定義(ACI)を入念に作る」。

パターン仕組み
プロンプト連鎖各LLM呼び出しが前の出力を処理する直列分解。
ルーティング入力を分類し、専門化された後続タスクへ振り分ける。関心の分離。
並列化複数LLMで同時処理し結果を集約。
オーケストレーター–ワーカー中央LLMが動的にタスク分解し、ワーカーへ委譲、結果を統合。動的ワークフローの原型。
評価者–最適化者一方が生成、もう一方が評価とフィードバックをループで返す。検証・敵対役の原型。
10

横断ベストプラクティス

どの手法を使うときも効く、Claude Code公式の実践集。手法を「動かす」ための潤滑油。

① 検証で締める ― 最重要

エージェントは「出来たように見えたら」止まります。検証手段がないと、あなた自身が検証ループになり、すべてのミスがあなたの気づき待ちになります。合否を返す仕組み（テスト、ビルドの終了コード、リンタ、フィクスチャとの差分、デザインとのスクリーンショット比較）を与えれば、ループは自律的に閉じます。締め方には段階があります。

締め方仕組み強さ
1プロンプト内「実装後にテストを走らせて直すまで反復して」と同じ指示に含める手軽
/goal 条件別の評価器が毎ターン再チェックし、満たすまで継続セッション越し
Stop フックスクリプトで決定的にゲート。通るまでターンを終わらせない確実
検証サブエージェント新鮮なモデルが結果を反証しに行く（作業者≠採点者）独立

そして主張ではなく証拠を出させること（テスト出力・実行コマンドと結果・スクリーンショット）。証拠の確認は自分で検証し直すより速く、見ていなかったセッションにも有効です。「検証できないなら出荷しない」。

② 具体的なコンテキストを与える

「ログインのバグを直して」ではなく「セッションタイムアウト後にログイン失敗。src/auth/ のトークン更新を確認し、再現する失敗テストを書いてから直して」。対象ファイル・状況・「直った」の定義を示すほど、修正の往復が減ります。@ でファイル参照、画像の貼り付け、cat error.log | claude でのパイプ入力も活用します。

③ 早めの・頻繁な軌道修正

  • Esc で途中停止（コンテキストは保持され、方向転換できる）。Esc Esc / /rewind で会話・コードを巻き戻し。
  • 同じ問題で2回以上修正したら、コンテキストは失敗アプローチで汚れている。/clear して、学んだことを盛り込んだより良い初期プロンプトで再開する方がほぼ常に勝る。

④ コンテキストを積極管理する

  • 無関係なタスクの間で /clear。長いセッションに無関係な履歴が溜まると性能が落ちる。
  • /compact <指示> で要約の焦点を制御。チェックポイントで任意地点に復元。
  • 調査はサブエージェントに委譲（「サブエージェントでXを調査して」）。大量のファイル読みを別の窓に隔離し、本筋の窓をきれいに保つ。

⑤ 敵対的レビューを一段挟む

無人で長く動かすほど、完了とみなす前の独立チェックが効きます。新鮮なサブエージェントは差分と基準だけを見て、生成過程の理屈に引きずられずに評価します。バンドルの /code-review スキルや、計画(PLAN.md)との突き合わせレビューが使えます。ただし「レビュー役は何かしら指摘するもの」。正しさや要件に関わるギャップだけを挙げさせ、過剰設計を避けます。Writer/Reviewer を別セッションに分ける、テスト作成役と実装役を分ける、なども有効です。

⚠ よくある失敗パターン（公式）
ごちゃ混ぜセッション（無関係な話題で窓を汚す→ /clear）、修正の無限ループ（2回失敗で /clear ＋良いプロンプト）、肥大化CLAUDE.md（容赦なく刈る）、信頼するが検証しないギャップ（必ず検証を付ける）、無限探索（スコープを絞るかサブエージェントへ）。

手法の使い分け

「規模・不確実性・期間・並列性」の4軸で選ぶ。多くの場合は複数を連結する。

タスクの性質推奨アプローチ（連結）コンテキスト負荷
タイポ・ログ追加・リネーム直接指示（計画なし）＋検証低
複数ファイルに及ぶ新機能プランモードで探索→計画（04）→ SPEC.md（05）→ 新セッションで実装→ 敵対的レビュー（10）中
不慣れなコードの改修「コードベースに質問」→ プランモード→ 具体的プロンプト（06・10）中
数時間〜数日のアプリ構築初期化役＋実装役のハーネス（08）＋進捗/ADR引き継ぎ（07）＋E2E自己検証高
数千ファイルの移行・全体監査動的ワークフロー（09）＋ ultracode ＋ auto モード＋独立検証高
探索的（正解が不明）あえて曖昧なプロンプトで反応を見る→ 軌道修正（10）低
✔ 通底する4原則（これだけ覚えれば良い）
①コンテキストは有限資源＝高シグナルの最小集合を保つ。②探索と実装を分ける＝間違った問題を解かない。③記憶は外部に書く＝窓を越えて継承する。④必ず検証で締める＝証拠を出させる。各手法はこの4つの具体化に過ぎません。

なお、本資料はモデルの「開発手法」に焦点を当てており、特定のモデルバージョンに依存しない原則を中心にまとめています。ツールの機能（動的ワークフロー等）やUIは更新が速いため、導入時は各公式ドキュメントの最新版をご確認ください。
