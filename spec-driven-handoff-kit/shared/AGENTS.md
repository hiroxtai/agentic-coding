# AGENTS.md

このリポジトリで作業する **すべての AI コーディングエージェント**（GitHub Copilot / Claude Code /
OpenAI Codex）への共通指示書です。各エージェントはこのファイルを起動時に読み込みます。
**すべてのタスクで必ず従ってください。**

> このリポジトリは「**仕様駆動 ＋ インタビュー深掘り ＋ ADR 引き継ぎ**」を統合した運用を採用しています。
> 記憶はセッションごとにリセットされる前提で、知識は会話ではなく **ファイル**（specs / memory-bank /
> .specify）に残します。

---

## 0. 黄金ルール（4原則）

1. **小さく始め、確認してから進む** — 不明点は勝手に仮定せず、インタビュー（grill）で詰める。
2. **仕様を真実の源にする** — コードの前に spec を確定し、spec から plan→tasks→実装へ降ろす。
3. **記憶は外部ファイルに** — 開始時に引き継ぎを読み、終了時に必ず書き戻す。
4. **検証で締める** — 「完了」の前にテスト/リント/ビルドを実行し、結果（証拠）を示す。

## 1. 標準ワークフロー（どのタスク/ストーリーでも同じ）

```
(0) 開始     : memory-bank/progress.md と architecture.md、関連する decisions/ を読む
(1) 深掘り   : grill-me で要件・制約・前提・代替案を洗い出す（曖昧なら必ず実施）
(2) 仕様     : specs/<NNNN-slug>/spec.md を確定（何を・なぜ。技術詳細は書かない）
(3) 計画     : specs/<NNNN-slug>/plan.md（スタック・構成・制約）。必要なら plan も grill する
(4) 分解     : specs/<NNNN-slug>/tasks.md（単独で実装・検証できる小タスク群）
(5) 実装     : tasks を1つずつ実装し、各タスクで検証する
(6) 記録     : 重要判断は decisions/ にADR追記。progress.md を更新
(7) 引き継ぎ : セッション終了時に update-handoff を実行（差分を提示）
```

- 仕様駆動の各ステップは **Spec Kit** のコマンド（`/speckit.constitution` `/speckit.specify`
  `/speckit.plan` `/speckit.tasks` `/speckit.implement`）で実行してよい。
- プロジェクトの不変原則は `.specify/memory/constitution.md`（憲法）に従う。

## 2. インタビュー深掘り（grill）

要件・計画が曖昧なときは、実装前に **grill-me**（相手を質問攻めにして意図・制約・隠れた前提・
代替案を引き出すスキル）を使う。

- 1問ずつ、推奨回答を添えて聞く。直前の答えを掘り切ってから横に移る（広さより深さ）。
- 「次の具体的アクション（コードを書く/コミットする）が可能になったら」終了し、結論を spec に反映する。

## 3. セッション引き継ぎ（最重要）

- **開始時**: `memory-bank/progress.md`・`architecture.md`・関連 `decisions/` を読み、現在地を把握。
- **決定の記録**: 重要な技術/設計判断は、文脈が新鮮なうちに `memory-bank/decisions/` へ
  **ADR を1決定1ファイルで追記**（連番）。過去のADRは編集せず、変更は **supersede** で。
- **終了時**: `memory-bank/progress.md` を更新（現在の状態 / 次の一手 / 最近の変更に1行）。
- 各エージェントのネイティブ記憶（Copilot Memory / Codex memories / Claude のメモリ）は補助。
  **確定情報は必ずこのリポジトリのファイルに**置く（ネイティブ記憶は失効しうるため）。

## 4. プロジェクト概要 / コマンド / 規約（各自記入）

- 概要: <このプロジェクトが何かを1〜2行で>
- セットアップ: `<例: npm install>`
- テスト: `<例: npm test>`（完了の前に必ず実行）
- リント/整形: `<例: npm run lint>`
- コード規約: <標準と異なる点だけ。例: ES Modules を使う / 名前付きエクスポート優先>

## 5. やってはいけないこと

- シークレットをコミットしない。`generated/` 等の自動生成物を手編集しない。
- 既存ADRの削除・改変（supersede で対応）。spec を無視した実装。
