01

全体像：2つのループ

「出す前に自分で直す」内側のループと、「指摘を学習して二度と出さない」外側のループ。

コーディングエージェントによるレビューは、2つのループで考えると整理できます。 ①セルフレビュー・ループはPRを出す前に自分の差分を直すもの。 ②指摘学習ループはPRで受けた指摘の傾向を恒久ルールに変え、同じ指摘を再発させないものです。

02

セルフレビューの型

人に見せる前にエージェントが「別人のレビュアー」として差分を批評し、自分で直す。

セルフレビューの肝は「作業者≠採点者」です。実装した同じ文脈で甘く見るのではなく、 新しい視点で差分と基準だけを見て批評させます。重大度（Important / Nit）を付けさせ、 Important は必ず修正してから検証で締めます。

03

Claude Code でのセルフレビュー

ローカルの /code-review が本命。GitHub App なしでセッション内で差分を批評・修正できる。

① ローカル /code-review（PR前の自己レビュー）

任意の Claude Code セッションで実行すると、いまの差分をレビューします（GitHub App 不要）。 正しさのバグに加え、再利用・簡素化・効率の改善も拾います。

# いまの差分をレビュー（指摘を表示）
/code-review

# レビューして、見つけた指摘を作業ツリーに自動修正
/code-review --fix

# 指摘をPRのインラインコメントとして投稿
/code-review --comment

# クラウドの深い多エージェントレビュー（ultrareview）→ 結果を適用
/code-review ultra --fix

# 特定のパス/PRを対象に。effort を上げると広く（不確実な指摘も）拾う
/code-review src/auth/  # 例

② /security-review（脆弱性特化）と CI

バンドル済みの /security-review で差分のセキュリティ分析。CI に載せるなら公式の GitHub Action（anthropics/claude-code-security-review）がPR差分をスキャンして コメントします（PRをブロックはしない＝既存フローを壊さない）。

③ レビュー基準のチューニング：REVIEW.md

レビューの挙動は2つのファイルで調整できます。学習した指摘ルールは REVIEW.md に蓄積します。

ファイル役割
CLAUDE.md全タスク共通のプロジェクト指示。レビューでは「新たに導入された違反」を Nit として指摘。
REVIEW.mdレビュー専用・最優先。全レビューエージェントの先頭に注入される。重大度の再定義、スキップ、リポジトリ固有チェック（例「新規APIルートに結合テスト必須」）、検証バーを書く。

重大度は 🔴 Important（マージ前に直すべきバグ）／🟡 Nit（任意）／🟣 Pre-existing（既存バグ）の3段階。各指摘には検証の根拠が付き、偽陽性は検証ステップで除去されます。

▶ 指示例（セッション内）
▷ あなた実装できた。PRを出す前に /code-review --fix でセルフレビューして、 Important は直して。直したらテストを通して結果を見せて。
Claude Code差分をレビューしました。Important 2件（トークン失効の競合、テナント未スコープのクエリ）を修正し、Nit 3件は要約に記載。npm test 緑です。差分を提示します。
04

GitHub Copilot でのセルフレビュー

VS Code の「Copilot code review」を未コミット変更に実行。2026年3月から agentic に。

Copilot は VS Code でローカルの未コミット変更（または選択範囲）をレビューできます。 2026年3月から agentic になり、リポジトリを探索して横断的な文脈を読んでから指摘します （旧来の行単位の浅いレビューから進化）。PR を出す前のセルフレビューに使えます。

レビュー観点の指定：instructions ファイル

レビューのルールは次に書きます（コーディング用の指示と同じファイル群を読む）。

  • .github/copilot-instructions.md … リポジトリ全体のレビュー観点（4000字制限。短く要点を）。
  • .github/instructions/*.instructions.md … applyTo でパス別のレビュー観点。

さらに、Copilot Memory がコードレビューで学んだ規約を citation つきで自動保存し、 次回のコーディング・レビューに適用します（後述の学習ループのネイティブ版）。

▶ 指示例（VS Code・agent mode）
▷ あなたこの未コミットの差分を、別人のレビュアーとしてレビューして。 正しさ・セキュリティ・テスト不足を重大度付きで指摘し、Important は直して。 AGENTS.md と .github/instructions のルールに反していないかも確認して。
Copilot（agent mode）関連ファイルを読みました。resetPassword でレート制限が要求エンドポイントにのみ適用され、確認エンドポイントが無防備です（Important）。修正し、テストを追加しました。ほか Nit 2件。

※ Copilot のマネージド「code review」をPR上で自動実行する設定（リポジトリ/組織のルールセット）も併用できます。

05

Codex・共通の型

Codex でも同じ。どのエージェントでも「実装とPRの間に1工程」を挟むのが本質。

OpenAI Codex でも、セッションで差分レビューを指示できます。基準は AGENTS.md に置けば 3エージェント共通で効きます（Codex は AGENTS.md を連鎖で読む）。レビュー専用ルールは REVIEW.md を参照させる運用が分かりやすいです。

◆ 共通の型：Step 5.5「セルフレビュー」
どのエージェントでも本質は同じ ―― 「実装 → セルフレビュー → 修正 → PR」を1つの工程にする。 統合ワークフローの Step 5.5 として、self-review スキル/プロンプトで呼び出します。 呼び方だけが違います（Claude/Copilot: /self-review、Codex: self-review スキル）。
06

指摘学習ループ

あなたが望む「指摘の傾向を学習して指示書/スキルに保持」を、3つの層で実現する。

3つの層

層仕組み性質
① ネイティブ自動Copilot Memory：レビューで学んだ規約を citation つきで自動保存し、コーディング/レビューに自動適用（面横断）。Claude は CLAUDE.md/REVIEW.md 違反をレビューが指摘。自動・楽だが揮発的
② ルールファイル
（手動・確定）		繰り返しの指摘を REVIEW.md（Claude）/ *.instructions.md（Copilot）/ AGENTS.md に明示ルール化。git管理で永続・レビュー可能。恒久・あなたの統合キット向き
③ 専用レビューア
（任意）		CodeRabbit「Learnings」（返信・却下から学習）、Greptile（過去レビューから学習）、Qodo（Rule System）。導入すれば学習を肩代わり
07

learn-from-reviews パイプライン

②を半自動化するスキル。マージ済みPRの指摘を集め、繰り返しをルール案にして提示する。

収集はGitHubなら gh で行います。スキルが裏でこんなコマンドを使うイメージです。

# 最近マージした自分のPRを列挙
gh pr list --state merged --author @me --limit 20 --json number,title

# 各PRのレビューコメントを取得
gh pr view <number> --comments
▶ 指示例
▷ あなた/learn-from-reviews 最近マージしたPRのレビュー指摘から、繰り返し出ている ものを REVIEW.md と AGENTS.md のルールに追記する案を出して。
エージェント直近15PRを分析。3回以上出た指摘: 「ログにメール/ユーザーIDを出している」「新規エンドポイントにテストが無い」。これらを REVIEW.md の「常にチェック」に追記する差分を提示します（承認後に反映、コードは変更しません）。

09

実践例（指示例つき）

「指摘 → 学習 → 再発しない」を、1つの流れで具体的に。

① PR前のセルフレビューで Important を潰す

▷ あなた/self-review
エージェント差分をレビュー: Important「再設定エンドポイントにレート制限が無い」を修正、テスト追加。Nit 2件は要約に記載。npm test 緑。→ PRを作成できます。

② PRで人間レビュアーが指摘 → 修正

レビュアー（人間）ログに user.email がそのまま出ています。PIIなので消してください。
▷ あなた指摘の通り、ログから user.email を除いて。マスクするかIDだけにして。

③ 後日、指摘を学習してルール化（再発防止）

▷ あなた/learn-from-reviews
エージェント直近のPRで「ログのPII」が3回指摘されています。REVIEW.md の「常にチェック」に 「ログにメール/ユーザーID/リクエストボディを出さない」 を追記する差分を提示します（承認後に反映）。

④ 次のタスク ― 同じ指摘はもう出ない

▷ あなた/self-review
エージェント（REVIEW.md のルールを適用）Important「logger.info(user.email) はPII出力。マスクが必要」を検出し、PRを出す前に修正しました。
◆ これが狙い
一度人間に指摘された「ログPII」が REVIEW.md のルールになり、次回は人間に見せる前に セルフレビューが捕まえる。レビュー往復が減り、レビュアーはより本質的な点に集中できます。
10

専用レビューア・注意点

学習機能つきの専用ツールを足す選択肢と、運用上の落とし穴。

ツール特徴（学習面）
CodeRabbitPRワークフローが充実。「Learnings」で返信・却下から学習し好みを記憶。path filters でノイズ調整。
Greptileフルコードベース文脈に強く、過去レビューから学習して精度が上がる。大規模・複雑な依存向き。
Qodoガバナンス志向。Rule System で組織横断のルールを定義・強制。
⚠ 注意点
  • 人間レビューの代替ではない：セルフレビューは「出す前に明らかな問題を潰す」もの。最終判断は人間。
  • マネージドCode Reviewは有料：Claude Code の GitHub App 版は Team/Enterprise・約$15-25/review。ローカル /code-review はセッション内で使える。
  • ルールは短く：Copilot instructions は4000字制限。長い REVIEW.md も効果が薄れる。learn-from-reviews で重複・陳腐化を整理する。
  • ノイズ調整：専用レビューアは path filters・learnings で「何を指摘するか」を絞らないと疲弊する。
