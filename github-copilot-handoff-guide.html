01

結論と全体像

Copilot は2026年、ネイティブの自動記憶を獲得した。これを土台に、git管理の指示とMCPで補強するのが最適。

GitHub Copilot のセッション間引き継ぎは、2026年に大きく変わりました。ネイティブの「Copilot Memory」が、明示指示なしでリポジトリの知識を学習・保存し、次セッションへ自動で引き継ぐようになったためです。これが本命の土台で、その上にgit管理のカスタム指示（恒久・確定的）を置き、必要ならMCPで外部メモリを足す、という3層構成が2026年6月時点の最適解です。

◆ 3行サマリー
①Copilot Memory を有効化（自動・リポジトリ単位・エージェント横断の記憶）。②`copilot-instructions.md` と `AGENTS.md` を整備（毎回必ず効く確定層）。③Memory Bank式の確定ファイルと MCP メモリを必要に応じて上乗せ。
◆ 前回までの資料との関係
汎用論（07章・引き継ぎ資料）のCopilot 特化版です。前回の「規約層／基盤層／組込み層」でいうと、Copilot Memory は組込み層がネイティブに強化されたもの、カスタム指示は規約層、MCPメモリは基盤層に対応します。
02

3つのエージェント面

Copilotの「エージェント」は3つあり、引き継ぎの効き方が少し違う。まずここを区別する。

03

Copilot Memory（ネイティブ自動記憶）

明示指示なしで、リポジトリの知識を学習・保存し、次セッションへ自動で引き継ぐ。Copilot 引き継ぎの本命。

Copilot Memory は、エージェントが作業しながらリポジトリの事実（コーディング規約・アーキテクチャ・ファイル横断の依存関係）を自動的に学習・保存し、次のセッション開始時にそれをプロンプトへ自動注入する仕組みです。ユーザーが明示的に指示しなくても、使うほど賢くなる累積的な知識ベースが育ちます。

項目内容
スコープリポジトリ単位に厳密限定。書き込み権限者の操作でのみ作成、読み取り権限者のタスクでのみ使用。ソースコードと同じくそのリポジトリ内に留まる。
有効期限約 28日で失効（使用・再確認されるとタイムスタンプが更新され延命）。
対応面coding agent・Copilot CLI・code review で先行提供。面をまたいで学びを共有。
状態公開プレビュー・オプトイン・有料プラン。2026年3月4日から Pro/Pro+ は既定ON。
有効化GitHub の Copilot 設定でオン/オフ。どこで学習させるかを自分で選べる。
✔ 実測された効果（GitHub の A/B テスト）
coding agent：PRマージ率 +7%（メモリ有り90% vs 無し83%）。code review：好意的フィードバック +2%（77% vs 75%）。いずれも統計的に高度に有意（p<0.00001）。「タスクを任せたときに望む結果が得られる頻度が上がる」ことを意味します。
04

Memoryの仕組み：citation検証

鍵は「citation（コード上の根拠位置）付きで保存し、使う前に実時間で検証する」こと。これで陳腐化を防ぐ。

記憶システムの本質的な難所は「保存した知識がコードの変化後も正しいままか」です。GitHub はオフラインの整理サービスではなく、「検索は難しいが検証は易しい」という性質を突いたジャストインタイム検証（just-in-time verification）を採用しました。記憶には必ずcitation（裏付けとなるコード位置）を添え、使う直前にそれを照合します。

記憶の作成は、エージェントが呼び出すツール（store_memory）として実装されています。保存される記憶は例えば次の形です。

{
  subject: "API version synchronization",
  fact: "API version must match between client SDK, server routes, and docs.",
  citations: ["src/client/sdk/constants.ts:12", "server/routes/api.go:8",
               "docs/api-reference.md:37"],
  reason: "同期が崩れると連携が壊れる。次回更新時に全箇所を揃えるため"
}
05

エージェント横断の記憶共有

あるエージェントの学びを、別のエージェントが使える。これが Copilot Memory の真価。

Copilot Memory は単一セッションだけでなくエージェント面をまたいで共有されます。あるエージェントが検証した知識を、別のエージェントが再利用できるため、開発ワークフロー全体で知識が蓄積します。GitHub の公式例がこの流れです。

▶ もう一つの効用：知識の平準化
熟練者のPRから学んだ規約を、新人のPRに対して code review が「この箇所も更新が必要」と指摘できます。チーム内のベテランの暗黙知が、自動的に新メンバーへ転移するわけです。
06

カスタム指示（手動・git管理層）

毎回確実に効く、決定的で監査可能な層。自動・揮発的な Memory を補完する“確定の真実の源”。

Copilot Memory が自動・確率的・失効ありなのに対し、カスタム指示はあなたが書いて git にコミットする確定的な層です。リポジトリにある指示ファイルは毎回読まれ、レビューもできます。Copilot は複数の形式に対応します。

ファイル役割いつ使う
copilot-instructions.mdリポジトリ全体の指示（ビルド/テストコマンド、規約、作法）。まず最初に置く土台
AGENTS.mdツール横断の標準。ルート＋ネスト可、最も近いものが優先。coding agent は2025年8月から対応。複数AIツール併用・モノレポ
*.instructions.md.github/instructions/ に置き、applyTo グロブでパス別に適用。領域ごとに規約が違うとき
*.prompt.md.github/prompts/ の再利用プロンプト。定型操作をコマンド化。「引き継ぎ更新」等の反復作業
◆ Memory と指示ファイルの役割分担
Memory＝自動・学習・揮発的（28日）で「気づき」を拾う。指示ファイル＝手動・確定・恒久で「必ず守らせる規約」を固定する。両者は競合せず補完関係。確定情報は必ず指示ファイル側に置くのが鉄則です。
07

Memory Bank を Copilot で

人が読めて git に乗る“確定の引き継ぎ”が欲しいなら、Memory Bank 方式を Copilot に載せる。

自動の Memory に加えて、人手で管理する明示的な引き継ぎが欲しい場合は、cline 由来の Memory Bank 方式を Copilot で再現するのが定番です。確定情報を Markdown に置き、copilot-instructions.md から「毎回読む」よう指示するだけです。

# 1) 確定の引き継ぎファイルを用意
memory-bank/
  ├─ architecture.md      # 構造・設計判断
  ├─ progress.md          # done / 残り / 既知の問題
  └─ decisions/           # ADR（なぜを追記式で蓄積）

# 2) .github/copilot-instructions.md に“起動時に読む”を明記
## Session handoff
- 作業開始時に必ず memory-bank/progress.md と architecture.md を読むこと
- 重要な設計判断をしたら decisions/ にADRを追記すること
- 区切りでは progress.md を「plan / done / files / next」で更新すること

# 3) 定型化：.github/prompts/update-handoff.prompt.md
# 「memory-bank を現在の状態で更新して」と呼ぶだけで実行
✔ Memory との使い分け
Memory（自動）は細かな気づきを拾い、Memory Bank（手動ファイル）は「人が意図的に残したい確定事項」を持ちます。Memory は28日で失効しますが、ファイルは git にある限り永続。重要な引き継ぎはファイル側に置くのが安全です。
08

MCPで外部メモリ接続

ネイティブMemoryで足りない・ツール横断で共有したいときは、MCPメモリサーバを足す。面によって制約が違う。

Copilot は MCP に対応したため、OpenMemory MCP・mem0・agentmemory などの外部メモリ基盤を接続できます。ただしエージェント面によって MCP の対応範囲が異なる点に注意が必要です。

▶ いつ MCP メモリを足すか
ネイティブ Memory はリポジトリ単位です。複数リポジトリ/複数ツールをまたいで同じ記憶を共有したい、または高度な検索や長期保持が欲しいときに、agent mode で OpenMemory（ローカル）等を重ねるのが妥当です。

10

注意点・使い分け

Copilot 固有の落とし穴と、目的別の早見表。

  • Memory は揮発的：約28日で失効し、自動・確率的でプレビュー段階。確定情報は必ず指示ファイル/Memory Bank に置く。
  • Memory はリポジトリ単位：横断共有はしない。複数リポジトリ/ツールで共有したいなら MCP メモリを検討。
  • coding agent の MCP 制約：tools のみ・resources/prompts 非対応・リモートOAuth MCP 非対応。クラウドでは外部メモリに頼り切らない。
  • プラン要件：Memory は有料プラン＋オプトイン（Pro/Pro+ は既定ON）。組織では管理設定の確認を。
やりたいこと使うもの
手間ゼロで自動的に文脈を引き継ぎたいCopilot Memory（ON にするだけ）
必ず守らせる規約・恒久情報を固定したいcopilot-instructions.md ／ AGENTS.md
人手で意図的に引き継ぎを残したいMemory Bank式ファイル ＋ prompt files
領域ごとに別ルールを当てたい（モノレポ）*.instructions.md（applyTo）
複数ツール/リポジトリで記憶を共有したいMCPメモリ（OpenMemory等・agent mode）
◆ ひとことで
Copilot の引き継ぎは 「自動の Copilot Memory」と「確定の指示ファイル」の二本立てが核。前者で手間なく賢くなり、後者で重要事項を固定する。MCP はその先の拡張です。
