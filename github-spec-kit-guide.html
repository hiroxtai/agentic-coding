01

Spec Kit とは

テンプレート・CLI・プロンプトを束ね、コーディングエージェントに「仕様駆動開発」を持ち込む公式OSSツールキット。

Spec Kit は、GitHub が2025年9月にオープンソースとして公開した、仕様駆動開発（Spec-Driven Development, SDD）を実践するためのツールキットです。テンプレート・CLI（specify）・一連のプロンプトをパッケージ化し、「コードではなく仕様を真実の源（source of truth）にする」開発スタイルを、お使いのコーディングエージェントの上で実現します。

作者は GitHub の Principal Product Manager, Den Delimarsky 氏。GitHub 自身が「これは実験（experiment）であり、本質は特定のツールではなくプロセスそのものにある」と位置づけ、だからこそOSS化しています。2026年時点でも 0.x 系として活発に開発が続いています。

◆ 前回資料との関係
これは「コーディングエージェント開発手法ガイド」05章 仕様駆動開発の GitHub 版の具体実装です。後述の「憲法（Constitution）」は、同ガイド07章の CLAUDE.md / AGENTS.md による永続コンテキストと同じ発想で理解できます。
02

なぜ生まれたか（vibe-coding問題）

「それらしいが動かない」コード。原因はエージェントの能力ではなく、こちらの接し方にあった。

コーディングエージェントが強力になるにつれ、あるパターンが目立つようになりました ―「目標を伝える → コードの塊が返る → でもよく見ると動かない、あるいは意図を外している」。コンパイルが通らない、問題の一部しか解けていない、選ぶはずのないスタックになっている…。GitHub はこれをエージェントの能力の問題ではなく、こちらの「接し方」の問題だと整理します。

◆ 発想の核
エージェントを「検索エンジン」のように扱うのをやめ、「文字通りに解釈するペアプログラマー」として、曖昧さのない指示を与える。そのために仕様を「書いて放置する静的文書」ではなく、プロジェクトと共に進化する“実行可能な生きた成果物”として捉え直す ― これが Spec Kit の出発点です。
03

中核思想：意図が真実の源

「コードが真実の源」から「意図（仕様）が真実の源」へ。AIが仕様を“実行可能”にしたことで成立する転換。

Spec Kit は、仕様とコードの力関係を反転させます。従来は「仕様は書いて脇に置き、コードこそが正」でした。Spec Kit では仕様が中心に居座り、実装・チェックリスト・タスク分解を駆動します。あなたの主な仕事は舵を取ること、書く作業の大半はエージェントが担います。

GitHub はこの転換を 「ドキュメントが重要になったからではなく、AIが仕様を実行可能にしたから」と説明します。仕様が自動的に動くコードへ変わるなら、仕様こそが「何が作られるか」を決める ― だから仕様が真実の源になる、という論理です。

04

仕組み：憲法＋4フェーズ

土台に「憲法」を置き、Specify → Plan → Tasks → Implement を明確なチェックポイントで進む。各フェーズが検証されるまで次へ進まない。

まず土台として Constitution（憲法） を置きます。/.specify/memory/constitution.md にプロジェクトの“交渉不可能な”原則・制約を記述し、以降の全フェーズがこれに従います。その上で、明確なチェックポイントを持つ4フェーズが進みます。鍵は「各フェーズが完全に検証されるまで次へ進まない」こと。

フェーズあなたが与えるものエージェントが生成するもの
Specify
仕様化		「何を・なぜ」作るかの高レベル説明。ユーザー体験・ジャーニー・成功の定義が主役（技術の話はしない）。詳細な仕様。「誰が使い、どんな問題を解き、どう操作し、どんな結果が重要か」を記述した生きた成果物。
Plan
計画		ここで初めて技術へ。スタック・アーキテクチャ・制約（標準技術、レガシー連携、コンプラ、性能目標、社内ドキュメント）。制約を尊重した包括的な技術計画。複数案を比較させることも可能。
Tasks
タスク分解		（指示は最小限）仕様と計画の確定。単独で実装・テスト可能な小タスク群。例：「認証を作る」ではなく「メール形式を検証する登録エンドポイントを作る」。AIにとってのTDD的な足場。
Implement
実装		タスクのレビューと承認。タスクを1つずつ（可能なら並列で）実装。あなたは千行の塊でなく焦点の絞られた差分をレビューする。
✔ なぜ4分割なのか
実装時、エージェントは「何を（仕様）・どう（計画）・今どれを（タスク）」すべて分かった状態でコードを書きます。だから推測がなく、レビューも小さな単位で済みます。
05

人間の役割：steer と verify

成果物はAIが書く。あなたの仕事は方向づけと検証 ― 「進む前に立ち止まって批評する」。

Spec Kit のプロセスには明示的なチェックポイントが組み込まれています。各フェーズの境目で、あなたは生成物を批評し、欠落やエッジケースを見つけ、先に進む前に軌道修正します。AIが成果物を生成し、あなたがそれが正しいことを保証する、という分業です。各チェックポイントで自問すべきことは次の通りです。

  • 仕様は本当に作りたいものを捉えているか？ ― ユーザー体験と成功条件にズレはないか。
  • 計画は現実の制約を考慮しているか？ ― スタック・性能・コンプラ・レガシー連携は妥当か。
  • AIが見落としたエッジケースや抜けはないか？ ― タスクの粒度と網羅性は十分か。
▶ 「steer（操舵）」と「verify（検証）」
あなたは細部を書く人ではなく舵を取る人であり、同時に各段階の品質ゲートでもあります。これは前回ガイド10章の「検証で締める」「早めの軌道修正」と同じ精神です。
06

導入とコマンド

CLI「specify」でプロジェクトを初期化し、エージェント内のスラッシュコマンドで各フェーズを駆動する。

1. CLI の導入とプロジェクト初期化

specify CLI を uvx 経由で実行し、プロジェクトの雛形と構造をセットアップします。

# specify CLI でプロジェクトを初期化
uvx --from git+https://github.com/github/spec-kit.git specify init <PROJECT_NAME>

# 使用エージェントを明示する場合（非対話実行では既定で Copilot）
specify init <PROJECT_NAME> --integration claude

初期化時、導入済みのエージェントが検出され、対話的に選択できます。

2. スラッシュコマンドで各フェーズを実行

設定が済むと、エージェント内で次のコマンドが使えます（新しい版では speckit. 名前空間付き）。これらが表示されれば設定成功です。

⚠ コマンド名について
初公開時は /specify /plan /tasks でしたが、現在は衝突回避のため /speckit.specify のように名前空間付きに変わっています。版によって異なるため、初期化後に表示されるコマンド一覧で確認してください。
07

対応エージェント

「お好みのAIツールで使える」のが売り。主要エージェントを中心に幅広く対応する。

Spec Kit は特定のエージェントに縛られません。CLI が導入済みのエージェントを検出し、テンプレートを各ツール向けに展開します。公式が中心に挙げるのは GitHub Copilot・Claude Code・Gemini CLI の3つです。

08

なぜ効くのか

LLMの基本特性に根ざす。「パターン補完は得意、読心術は不可」への正攻法。

この手法が「ただプロンプトする」より優れる理由は、LLMの本質にあります。パターン補完は卓越しているが、心は読めない。「写真共有を追加して」のような曖昧な指示は、明文化されていない数千の要件をAIに推測させ、その一部は必ず外れます ― しかも実装の深部に入るまで気づけません。仕様・計画・タスクを前もって与えれば、「何を・どう・どの順で」作るかが確定し、推測が消えます。

だからこそ技術スタックを問わず効きます（Python でも JavaScript でも Go でも、課題は「意図を動くコードに翻訳すること」で共通）。仕様が意図を、計画が技術判断を、タスクが実装単位を、それぞれ担います。

◆ 組織にとっての価値
セキュリティ方針・コンプラ規則・デザインシステム・連携要件 ― こうした要件は往々にして「誰かの頭の中／読まれないWiki／流れていくSlack」に散っています。Spec Kit ではそれらが仕様と計画に集約され、AIが実際に使える形になります。セキュリティ要件は後付けでなく「初日から仕様に焼き込まれる」ものになります。

10

位置づけ・注意点・将来

「ツールより本質はプロセス」。実験段階ゆえの課題と、向かう先。

GitHub は Spec Kit を「その移行を現実にするための実験」と位置づけ、「本当の革新は特定のツールや企業ではなくプロセスそのものにある」としてOSS化しました。一方で実験段階ゆえの課題も率直に挙げられています。

  • 多数の Markdown を大規模に管理する難しさ ― 仕様・タスクのファイルが増えると整理・集中が課題に。
  • ワークフローの体験 ― 大量のテキストを読むのは退屈になりがちで、より魅力的・実用的にする余地がある。
  • VS Code 統合や複数実装の比較・差分 ― 今後の探求テーマとして公式が言及。
  • 既存システムでの機能追加を機能させるには、高度なコンテキストエンジニアリングの併用が必要になる場合がある。
◆ 大局：意図が真実の源へ
Spec Kit が体現するのは 「コードが真実の源」→「意図（仕様）が真実の源」という移行です。これは前回ガイドで述べたコンテキストエンジニアリング（必要十分な高シグナル情報をエージェントに与える）と表裏一体であり、GitHub も「SDDとコンテキストエンジニアリングの組み合わせ」を次のテーマに挙げています。
